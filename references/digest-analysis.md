# byteworker · digest 细则 —— 参与方立场分析 与 思路视角沉淀

> 由 `SKILL.md`「digest」一节路由到这里。本次 digest 若会产出 `event` 的「参与方立场分析」
> 章节、或要往 `project`/`area` 的「思路与视角」章节追加条目,动手前必读本文件。
> schema(这两个 body 章节的归属、字段)以 `DESIGN.md` §4.2 为准;本文件锁定的是**怎么写**。

byteworker 不只存字面结论,还消化两层**主观信息**:各参与方的「立场」、以及第一方的「视角」。
两者互补 —— **立场分析**是从会议发言**推断**出来的(observed);**思路视角**多是有人
**直接陈述**的(stated)。这两层最容易带偏,准则的核心是一句话:**严守证据、绝不臆造**。

## 一、参与方立场分析 —— 写进 `event`,沉淀进 `person`

`event` 节点除字面结论外,需对关键参与方做立场分析,并把结论同步沉淀进对应 `person` 节点的
「立场 / 利益 / 动机」章节。规则:

- **三个维度**:① 立场 / 倾向 —— 对所讨论决策的位置;② 利益 / 动机 —— ta 在为谁、为什么
  优化(代表哪方、核心诉求);③ 对决策的态度 —— 支持 / 反对 / 中立 / 有条件支持。
- **必须基于证据**:每条结论可追溯到 ta 的具体发言或行为,关键处引用原话或注明出处。
- **区分观察与推断**:【观察】= ta 明确说了 / 做了;【推断】= 基于证据的合理推测。
- **不臆测**:证据不足时写「证据有限」或略过,**不做无证据的发散猜测**;宁可少写、写得
  保守,也不编造动机。

> 目的:让知识库能回答「某人对某决策什么态度、背后动机是什么」,同时不沦为无根据的揣测。

## 二、思路与视角 —— 写进 `project` / `area` 的「思路与视角」章节

`project` / `area` 节点的「思路与视角」章节承载**第一方、主观、被主动输入**的观点 ——
使用者本人的工作思路、主管的想法 / 方向、合作同事的判断。它与第一节互补:第一节是从发言
**推断**(observed),本节多为**直接陈述**(stated)。

- **章节形态 = 带日期、带作者、只追加的日志**(同「关键进展」),不就地改写。当前思路 =
  最新一条;旧条目保留为可见的演进轨迹,过期太多可滚入「历史」。
- **条目格式**:`- 【主张|意图】<作者:节点 id 或姓名> · <YYYY-MM-DD> —— <内容>`,新条目在前。
  - 作者尽量写成 `person-` 节点 id(触发自动连边);无对应节点则写姓名。
  - 无明确作者时默认 = 知识库主人(使用者本人)。
- **出处标记**:【主张】= 某人主张 / 认为如此;【意图】= 某人想做 / 打算做。与第一节的
  【观察】/【推断】并列 —— 第一方陈述用【主张】/【意图】,从发言推断仍用【推断】。
- **绝不硬化为事实**:这是最易带偏的一层。synthesize / 答复时,【主张】/【意图】永远呈现为
  「某人的判断 / 意图」,不得当成客观结论;与客观信息冲突时主动提示。
- **更老化更快**:思路比 decision 过期快 —— 答复时优先用最新条目,引用旧条目时提示
  「这是 N 周前的思路,可能已变」。

## 分工与边界

- **本节两层的标记体系并列**:【观察】/【推断】(立场分析,推断)↔【主张】/【意图】(思路视角,陈述)。
- **节点「思路与视角」 vs `context.md`**:本章节是**挂在具体 `project`/`area` 上**的观点;
  `context.md`(DESIGN.md §10)是**跨主题**的工作底色。digest 时,针对具体项目的思路落到
  节点本章节;跨主题、不挂某个项目的工作底色不进节点 —— 留给使用者维护 `context.md`。
- **会议簇里的素材区分**(详见 `references/digest-meeting.md`):投屏文档是主讲人**预设的
  框架** → 进「思路与视角」的【主张】/【意图】素材;妙记逐字稿是各方**真实发言** →
  「参与方立场分析」的证据。两者别混。
